### O Que Vem Por Aí ?
É comum que códigos de ciência de dados funcionem em um ambiente, mas falhem em outro por diferenças em dependências ou sistemas. O Docker resolve esse problema ao empacotar aplicações e todas as suas dependências em contêineres isolados e padronizados, garantindo o mesmo funcionamento em diferentes máquinas.
Isso facilita a reprodut ibilidade, a colaboração entre equipes e a portabilidade entre ambientes de desenvolvimento e produção. Nesta aula, vamos entender como criar imagens Docker, executar contêineres, aplicar boas práticas e analisar casos reais no contexto da ciência de dados .

Contêin eres vs. Máquinas Virtuais: virtualização leve
Uma dúvida comum ao iniciar é: “Contêiner não é só uma VM?” . Embora ambos providenciem isolamento, a forma como o fazem é bem diferente. Máquinas virtuais (VMs) tradicionalmente usam hipervisores para virtuali zar o hardware inteiro :
cada VM roda um sistema operacional completo (kernel + user space) em cima de um hardware virtualizado, o que consome bastante recurso.
Já um contêiner Docker realiza uma virtualização a nível de sistema operacional (OS-level virtu alization ), compartilhando o mesmo kernel do host, porém isolando processos, sistema de arquivos e rede através de primitivas do próprio OS
(no caso do Docker, primitivas do kernel Linux). Na prática, um contêiner pode ser visto como um processo isolado rodando no host, com sua própria visão do sistema
(como se tivesse seu próprio root filesystem, seus próprios processos PID 1, sua interface de rede independente etc.).
Isso faz com que contêineres sejam muito mais leves – não carregam um sistema operacional inteiro dentro de si, apenas usam partes necessárias,
aproveitando o kernel já em execução. Uma VM leva minutos para iniciar; um contêiner sobe em segundos (ou menos) devido a essa leveza. Em termos de overhead, contêineres gastam bem menos CPU e memória para o isolamento.

Como bem resumiu Vinícius Victorino (2022) , uma VM tradicional isola executando outro sistema operacional completo, consumindo muito mais recursos,
enquanto contêineres isolam aplicações como processos do mesmo SO do host, de forma muito mais enxuta .
Tecnicamente, o Docker depende de duas funcionalidades -chave do Linux:
namespaces e cgroups (grupos de controle). Namespaces implementam o isolamento: cada contêiner ganha um conjunto de namespaces próprios – por exemplo, PID name space isolando sua árvore de processos, NET namespace isolando seus dispositivos de rede virtuais, MNT namespace isolando seu sistema de arquivos montado etc.
Isso garante que um processo dentro do contêiner tenha PID 1 no contexto dele (como se fosse ini t do “mini -sistema”), mas na verdade é apenas um processo normal do ponto de vista do kernel global. Já os cgroups cuidam do gerenciamento de recursos, permitindo impor limites e quotas de CPU, memória, I/O etc. para grupos de processos. O Docker cria cgro ups para cada contêiner conforme necessário – por exemplo, se definirmos que um contêiner pode usar no máximo 2
GB de RAM, o kernel aplicará esse cgroup limitando a alocação de memória daquele grupo de processos.
Assim, mesmo que dentro do contêiner haja um pânico de consumo de memória, ele não derrubará o host; o kernel pode inclusive matar processos do contêiner (gera um OOM kill) se excederem o limite, protegendo o sistema. Por padrão, se não estipularmos nada, o contêiner pode usar recursos conforme disponível no host (ou seja, sem limites específicos além do compartilhamento padrão do SO). Mas o Docker nos dá a opção de setar flags no docker run para controle fino – -m/--memory para limite de RAM, --cpus ou -c (shares) para controlar
CPU etc., com suporte do kernel Linux através dos cgroups para fazer cumprir esses limites.
Em resumo, contêineres são implementados como ambientes isolados no espaço do usuário , usando o mesmo kernel do host. Isso explica por que contêineres Linux rodam nativamente só em hosts Linux (ou em VMs Linux se o host for Windows/Mac, o que ferramentas como Docker Desktop fazem nos bastidores).
No entanto, existem contêineres Windows análogos (embora menos populares em
DS) que usam kernel W indows e funcionam apenas em hosts Windows.

Gerenciamento de recursos: CPU, memória e GPU nos contêineres
Para aplicações de ciência de dados , desempenho é importante – treinar um modelo pesado envolve muita CPU/GPU e memória. Como os contêineres lidam com isso? Já vimos que, por padrão, um contêiner pode usar todo recurso disponível, competindo de forma “justa” com outros processos do host. O Docker permite limitar e reservar recursos via cgroups: por exemplo, podemos rodar docker run --memory=4g --cpus=2 para restringir a 4 GB de RAM e ~2 CPUs o contêiner.
Esses limites duros garantem que, mesmo que o código dentro do contêiner tente usar mais (threads extras, mallocs enormes), o kernel irá contê -lo – evitando travar o host por falta de memória, por exemplo.
Em cenários de multiusuário ou vários contêineres simultâneos (como vários experimentos rodando em paralelo em um servidor), isso é essen cial para isolar consumo. Também há como definir soft limits e reservas – por exemplo, --memory -
reservation para dar preferência e CPU shares para peso relativo de CPU – quando vários contêineres competem, o scheduler pode dividir o tempo de acordo com esses pesos.
Um recurso especial é a GPU . Contêineres inicialmente não tinham suporte direto a GPU (porque a GPU não era visível no namespace de dispositivos), mas hoje é possível habilitar. Para usar GPUs NVIDIA, por exemplo, instala -se o NVIDIA
Container T oolkit no host e executa -se o contêiner com --gpus (ex.: docker run --
gpus all ... nvidia/cuda:12.2 -python3.10 ...). Isso permite que o contêiner acesse as placas de vídeo do host para treinar modelos de Deep Learning, por exemplo.
Internamente, o Docker m onta os drivers necessários no contêiner e autoriza acesso aos dispositivos /dev/nvidia*.
Assim, conseguimos isolamento de software sem perder aceleração por hardware . Em cenários de clusters, também há orquestradores (como Kubernetes

Do ponto de vista de sistemas operacionais , o Docker se apoia em mecanismos de escalonamento e alocação do kernel. Contêineres repartem CPU de acordo com o scheduler Linux normal (CFS), a não ser que usemos afinidades ou limites. Já a memória, como mencionado, pode ser restringida. Se um contêiner consome memória descontroladamente e o host esgota RAM, o kernel Linux pode acionar o OOM Killer – frequentemente ele irá matar o processo do contêiner problemático antes de matar o próprio Docker ou outros serviços do sistema. Isso é bom: mantém a estabilidade do host. Podemos sintonizar esse comportament o (há flags avançadas como --oom-kill-disable, não recomendadas sem -m, pois desabilitar o OOM killer sem limitar memória pode levar o host ao colapso).
Em resumo, o Docker oferece controle refinado de recursos , o que é útil em cenários de ciência de dado s compartilhados (ex.: vários cientistas rodando experimentos no mesmo servidor – cada um no seu container limitado para não atropelar o outro). Também é útil para simular ambientes de produção com menos recursos: por exemplo, testar se seu pipeline ML deg radaria em uma máquina com apenas 1 CPU, limitando seu container a 1 CPU para ver o efeito.
Vale mencionar que contêineres herdam o kernel do host . Portanto , certos comportamentos são idênticos ao host – por exemplo, um contêiner Linux terá a mesma gestão de memória virtual do host (incluindo estratégias de paginação).
Técnicas de alto desempenho, como uso de huge pages para memória compartilhada em treino de modelos, precisam ser suportadas pelo kernel host para refletir no contêiner. No entanto, o isolame nto garante que dentro do contêiner a aplicação “enxerga” apenas os recursos alocados a ela. Em termos de implementação de baixo nível , além de cgroups e namespaces,
o Docker usa Union File Systems para as imagens – cada camada do Dockerfile é uma sobreposição de sistema de arquivos somente -leitura, com o contêiner ativo possuindo uma camada de leitura -escrita no topo. Isso possibilita o reuso eficiente de camadas entre contêineres e rollback rápido, mas significa que a escrita intensiva na camada superior pode ter pequena penalização. Para dados pesados (como escrita de checkpoint s de modelo), é recomendável usar volumes montados (que escrevem diretamente no host ou em disco externo) para performance nativa.
Escalabilidade e distribuição de contêineres
Um único contêiner rodando em uma máquina já resolve o “funciona em qualquer lug ar”; mas e se quisermos lidar com grandes cargas de trabalho ou alta disponibilidade ? Aí entra a possibilidade de escalar horizontalmente nossos ambientes Docker. Na prática, escalar contêineres significa rodar múltiplas instâncias de um mesmo contêiner (r éplicas ) distribuídas em um ou vários hosts e coordená -las. Diferentemente de processos tradicionais, contêineres têm a vantagem de serem facilmente replicáveis: como cada instância carrega todas as dependências, você pode levantar 10 contêineres idênticos da sua aplicação de ML sem temer diferenças entre eles.
Isso é muito usado em produção para servir modelos em larga escala – por exemplo, se um modelo de recomendação precisa servir milhões de requisições,
você coloca várias réplicas do serviço em contêi neres atrás de um balanceador de carga. Ferramentas de orquestração como Kubernetes automatizam esse processo de escalonamento.
O Kubernetes (K8s) é hoje a plataforma mais adotada para gerenciar contêineres em cluster. Conceitos -chave: você define um deplo yment para sua aplicação contêinerizada , especificando quantas réplicas quer e o K8s se encarrega de agendar esses contêineres ( pods ) nos nós disponíveis, monitorando e mantendo o estado desejado. Ele usa um escalonador (scheduler) interno que decide em qu al máquina cada contêiner roda, considerando recursos disponíveis, afinidades, etc. –

Para dimensionamento dinâmico, o Horizontal Pod Autoscaler (HPA) do K8s pode aumentar ou reduzir o número de contêineres de acordo com métricas de uso
(CPU, memória ou métricas customizadas). Por exemplo, se o uso de CPU médio dos pods passar de 80%, o HPA pode subir mais instâncias; se cair abaixo de um limiar, remover algum as, tentando acompanhar a demanda. Já o Vertical Pod
Autoscaler (VPA) ajusta recursos de cada contêiner (dando mais CPU/memória para um pod conforme necessidade) – mas para cargas de trabalho de ciência de dados,
escalar horizontalmente costuma ser mais eficiente quando possível (treinamentos distribuídos etc.), salvo casos de componentes não paralelizáveis que precisam de mais memória. Em ce nários de big data , contêineres são usados também para orquestrar frameworks c omo Spark em clusters – cada executor Spark pode rodar dentro de um contêiner, facilitando isolamento e reutilização de recursos no cluster.
Um conceito importante é o de balance amento de carga . Quando temos múltiplos contêineres replicando um serviço (por exemplo, vários servidores de um modelo ML), precisamos distribuir as requisições entre eles. Ferramentas de orquestração costumam fornecer service discovery e load balancing . No Docker standalone, existe o Docker Swarm que tem recursos nativos de distribuir carga entre contêineres em cluster. No Kubernetes, um Service abstrai um conjunto de pods e pode rodar um proxy/balanceador (como kube -proxy) para rotear pedidos para os pods disponíveis. Os algoritmos de roteamento geralmente são de rodízio simples ( round -robin ) ou baseados em acaso, às vezes considerando o peso. O essencial é que contêineres tornam a escala horizontal trivial: se um pod cair, outro pode substituí -lo rapidame nte (K8s cuida de reiniciar) e se precisarmos dobrar throughput, simplesmente dobramos o número de contêineres.
Para o processamento distribuído de dados , contêineres podem encapsular componentes de frameworks como Hadoop, Spark, Dask. Um exemplo: você pod e ter um cluster Kubernetes rodando contêineres Spark – o driver e executors são contêineres separados. Nesses cenários, surge a necessidade de orquestrar centenas ou milhares de contêineres , onde surgem desafios de granularidade (quão grande cada contêiner deve ser e m termos de tarefas?), eficiência de escalonamento e rede. Protocolos como MPI para treinamento distribuído também podem rodar em contêineres – cada nó MPI contido , mas comunicando via rede do cluster.
Do ponto de vista algorítmico, escalar horizontalmente tem limites teóricos dados pela Lei de Amdahl : se uma porção P do seu workload é paralelizável, em teoria escalar para N instâncias reduz o tempo nessa parte em até N vezes, mas sempre haverá a parcela sequencial (1 -P) que permanece como gargalo. Por isso ,
adicionar contêineres indefinidamente dá retornos de crescentes se o problema não paraleliza bem. Cientistas de dados devem estar cientes disso ao arquitetar soluções: às vezes melhorar o algoritmo (aumentar P) vale mais do que só jogar mais contêineres.
Ainda assim, contêineres facilitam alcançar o speedup ideal porque removem variáveis de ambiente – cada instância paralela é idêntica. Em clusters modernos, a sobrecarga de coordenação entre contêineres (latência de rede etc.) também influencia; ferrament as como Kubernetes introduzem pequena latência ao escalar
(HPA não é instantâneo, leva segundos para spawn pods). Para demandas ultrarrápidas , podem ser necessárias estratégias de predictive scaling (escalar antes do pico previsto) ou usar arquiteturas ser verless complementares.

Por exemplo, um pesquisador pode publicar a imagem do seu experimento de deep learning; outro em outro país pode baixar e rodar exatamente aquele ambiente. Engenharia de software aplicada à ciência de dados com Docker
Do ponto de vista de melhores práticas de engenharia, o Docker traz para data science conceitos maduros de DevOps e infraestrutura como código . Com ele,
o ambiente de execução se torna código (o Dockerfile e arquivos de configuração),
que pode ser versionado no Git junto com o código -fonte do projeto. Isso significa que mudanças de ambient e (atualizar uma biblioteca, trocar versão de Python) são rastreáveis e reversíveis como qualquer mudança de código. Essa abordagem disciplinada diminui bastante a famosa frase “na minha máquina tinha pacote X que agora não lembro a versão” – tudo fica doc umentado declarativamente no
Dockerfile.
Além disso, ele habilita CI/CD efetivo para projetos de ciência de dados:
podemos ter pipelines automatizados que constroem a imagem Docker e executam testes dentro do contêiner sempre que o novo código é commitado , garantindo que não faltam dependências e que o código roda em um ambiente limpo. Essa filosofia segue o conceito de infraestrutura imutável : em vez de configurar servidores à mão,
você recria o ambiente do zero conforme a especificação sempre que necessá rio,
obtendo consistência.
Outro ponto de aproximação com a engenharia de software é a modularidade e a reutilização . Imagens Docker podem ser compostas ou estendidas – por exemplo, você pode construir uma imagem base com ferramentas comuns de ciência de dados (Python + SciPy stack) e depois derivar imagens específicas de projetos adicionando apenas o código e pacotes exclusivos. Isso lembra os princípios de orientação a objetos (herança) ou de componentes reutilizáveis. Times de ciência de dados grandes f requentemente mantêm imagens base padronizadas
(com versões aprovadas de certas libs, configurações de proxy etc.), sobre as quais cada novo projeto se baseia. Isso economiza tempo e assegura que todos partem de um ambiente comum testado.
