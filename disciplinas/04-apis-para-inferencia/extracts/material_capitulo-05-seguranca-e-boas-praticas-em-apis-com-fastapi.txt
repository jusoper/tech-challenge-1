Autenticação e Autorização (Tokens JWT)
Em segurança de APIs, autenticação é o processo de verificar quem é o usuário ou sistema que faz a requisição, enquanto a autorização determina o que esse usuário pode fazer. Em aplicações web modernas, especialmente APIs REST stateless, é comum usar token s de acesso para essas tarefas, em vez de sessões tradicionais.
Um dos padrões mais difundidos é o JWT (JSON Web Token) , um token em formato JSON, assinado criptograficamente, que carrega claims como usuário,
permissões, validade etc. que o servidor pode verificar sem precisar de armazenamento de sessão.

Rate Limiting - Protegendo contra Abusos
Mesmo em rotas protegidas, sem um limite de requisições, um atacante pode derrubar a nossa API com milhares de chamadas ou até mesmo consumir uma quantidade absurda de recursos GPU/CPU, gerando custos extras na cloud.
Para nos ajudar com isso , o Rate Limiting limita X requisições por Y segundos por cliente. Por exemplo, um usuário do mesmo IP pode fazer no máximo 10 requisições por minuto.

CORS: Controlando o Acesso de Origens Cruzadas
APIs podem ser consumidas por diversos clientes, como aplicações web em navegadores, que impõem a política de Same -Origin (requisições AJAX/XHR/fetch apenas para o mesmo do mínio de origem). Para permitir que origens diferentes acessem a API, é necessário habilitar o CORS (Cross -Origin Resource Sharing).
O CORS é um padrão que permite ao servidor informar ao navegador quais origens externas (domínios) podem acessar seus recur sos. Isso é feito através de cabeçalhos HTTP específicos, como Access -Control -Allow -Origin .
Quando habilitar CORS?
• Necessário: se sua API for acessada por um frontend (aplicação web)
hospedado em um domínio diferente (ex .: API em api.minhaempresa.com e frontend em app.minhaempresa.com).
• Opcional/Desnecessário: se a API for usada apenas em comunicação server -to-server (sem navegador). Desabilitar o CORS nesse cenário aumenta a segurança, prevenindo chamadas de scripts em páginas web.
Segurança e Boas Práticas em APIs
 O FastAPI facilita a co nfiguração do CORS usando um middleware dedicado.
Você define explicitamente as origens, métodos HTTP e cabeçalhos permitidos .
Para permitir requisições de um domínio de produção e de um ambiente de desenvolvimento local ( localhost ), utilizando os métodos GET e POST , o código seria:
from fastapi.middleware.cors import CORSMiddleware from fastapi import FastAPI app = FastAPI()
# Lista de domínios permitidos
origins_permitidas = [
 "https://app.minhaempresa.com",
 "http://localhost:3000", # Ex: ambiente local de testes do frontend
]
app.add_middleware(
 CORSMiddleware,
 allow_origins=origins_permitidas,
 allow_credentials=True, # Permite envio de cookies/autenticação allow_methods=["GET", "POS T"], # Métodos HTTP autorizados allow_headers=["*"], 

# Quais cabeçalhos podem ser enviados (o "*" permite todos)
)
Essa configuração fará com que o FastAPI inclua os cabeçalhos CORS necessários nas respostas, autorizando o acesso das origens esp ecificadas.

 Boas Práticas Gerais em APIs de ML
Além dos pontos específicos que implementamos, existem várias práticas gerais de segurança e qualidade que devem ser adotas ao construir APIs (seja para
ML como para outros fins).
• Logs e Monitoramento: registre as ações da API para detectar erros e ataques. Grave detalhes da requisição (caminho, método, status) sem dados sigilosos (senhas, tokens). Use logs estruturados (JSON) para facilitar a análise. Monitore falhas frequentes.
• Comunicação Segura (HTTPS): sempre use HTTPS em produção. Ele criptografa o tráfego para proteger dados sensíveis. O certificado é fácil de obter. Use HTTP só para testes internos.
• Gerenciamento de Segredos: nunca deixe senhas ou chaves de acesso no código. Use variáveis de ambiente, arquivos de configuração protegidos
(.env não versionado ) ou serviços de gerenciamento de segredos. Não exponha informações sensíveis em URLs ou mensagens de erro.
• Atualização de Dependências: mantenha suas bibliotecas e frameworks sempre atualizados para corrigir falhas de segurança. Use ferramentas de varredura. A segurança exige atualizações rápidas, mesmo em sistemas funcionando.
• Princípio do Menor Privilégio: dê a cada parte do sistema apenas as permissões necessárias. Ex: usuário de banco de dados só com acesso de leitura, se for o bastante. Não rode a aplicação como root desnecessariamente.
• Testes de Segurança e Hardening: inclua testes de segurança (com ferram entas como OWASP Zap) no desenvolvimento. Verifique a proteção de endpoints de administração. Em produção, desabilite recursos desnecessários (como documentação automática ou listagem de diretórios)
e reforce cabeçalhos HTTP de resposta.

 Considerações Esse nciais para APIs de Inferência de ML em Produção
Levar uma API de inferência de Machine Learning (ML) à produção exige uma visão que vai além do código, focando na robustez e resiliência da infraestrutura. Os pilares de um serviço de sucesso são: desempenh o e resiliência da infraestrutura ;
segurança e operação contínua .

Em produção, a operação exige atenção a outros fatores cruciais:
• Observabilidade: implementação de logs centralizados, métricas de performance e traços para diagnóstico e auditoria.
• Segurança em Profundidade: múltiplas camadas de proteção:
autenticação na API, Web Application Firewall (WAF) e monitoramento ativo.
• Controle de Custos: uso de rate limiting para prevenir abusos e escalabilidade elástica para evitar gastos com capacidade ociosa.
• Conformidade: seguir as recomendações de segurança da comunidade,
como o OWASP API Security Top 10 (ex .: mitigação de autenticação quebrada, vazamento de dados).
